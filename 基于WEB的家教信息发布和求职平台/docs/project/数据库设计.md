# 数据库设计

## 1. 设计原则

数据库围绕用户、教员资料、家教需求、接单申请、订单和审核记录六类核心对象设计。用户表统一存储家长、教员和管理员账号，其他业务表通过外键关联用户，保证流程可追踪。

## 2. 核心表

- `user_account`：用户账号表，包含用户名、密码、姓名、电话、角色和状态。
- `tutor_profile`：教员认证资料表，关联教员账号，记录学校、专业、擅长科目、教学经验、服务方式、文字简历、简历附件元数据和审核状态。
- `tutor_demand`：家教需求表，关联家长账号，记录科目、年级、地点、预算、时间和需求状态。
- `tutor_application`：接单申请表，关联需求与教员，记录申请说明和处理状态。
- `tutor_order`：订单表，关联需求、申请、家长和教员，记录服务状态。
- `audit_record`：审核记录表，记录管理员对资料、需求、申请的审核行为。

## 3. 主要状态

- 教员认证状态：`PENDING`、`APPROVED`、`REJECTED`。
- 需求状态：`PENDING_REVIEW`、`OPEN`、`MATCHED`、`CLOSED`、`REJECTED`。
- 申请状态：`SUBMITTED`、`ACCEPTED`、`REJECTED`、`CANCELLED`。
- 订单状态：`ACTIVE`、`FINISHED`、`CANCELLED`。

## 4. 关系说明

- 一个家长可以发布多个家教需求。
- 一个教员拥有一条认证资料。
- 一条教员资料同时支持文字简历和附件简历两种录入方式，附件文件路径保存在资料表中。
- 一个需求可以收到多个教员申请。
- 一个被确认的申请生成一个订单。
- 审核记录保留管理员对关键业务对象的处理轨迹。
