# 1 引言

本课题面向新闻信息持续增长背景下的热点发现与趋势研判需求，围绕“新闻数据从哪里来、如何处理、怎样分析、如何展示”四个核心问题，设计并实现一个基于 Spark 的新闻热度监测系统。中国互联网络信息中心发布的第 55 次《中国互联网络发展状况统计报告》指出，我国互联网基础资源和移动互联网接入流量仍保持持续发展态势，互联网应用已经深度嵌入社会生产和公众信息获取过程[1]。在这种背景下，新闻信息呈现来源多样、更新频繁、转载重复和主题变化快等特点，传统依靠人工浏览、手工整理和单一关键词检索的方式难以及时发现热点变化。

本研究并不将可视化页面视为唯一目标，而是将数据采集质量、清洗去重质量和分析结果可解释性作为核心主线，强调在单机实验环境下完成新闻采集、批处理分析、结果展示与数据导出的完整链路。从应用场景来看，新闻热度监测系统可为媒体编辑、舆情分析人员和行业观察者提供热点事件发现、关键词趋势识别、情感倾向分析和专题事件聚类等基础能力。该课题兼具工程实现价值与研究写作价值，既能够体现大数据处理技术在文本分析场景中的应用特点，也便于围绕系统设计、实现与测试过程展开系统论述。

## 1.1 研究背景

互联网媒体、门户网站、RSS 订阅源和开放新闻数据集持续产生大量新闻文本，不同来源之间在字段结构、发布时间格式、正文完整度和页面噪声方面存在明显差异。与此同时，热点新闻往往会被多个媒体反复转载，同一事件可能以相近标题、不同摘要或不同链接形式出现。如果系统仅按照原始新闻数量进行统计，重复转载内容会放大某些事件的热度，导致热点榜单和趋势结果失真。因此，新闻热度监测的关键不只是把新闻“抓下来”，还要对新闻数据进行清洗、标准化和去重处理。

从计算方法看，新闻文本分析涉及关键词统计、时间窗口聚合、来源分布统计、主题或事件聚类以及情感倾向识别等环节。文本情感分析研究通常将任务划分为情感信息抽取、情感分类以及情感检索和归纳等方向，其在舆情监测、用户观点分析和文本挖掘中具有广泛应用[2]。文本倾向性分析综述研究也指出，文本倾向性分析需要经过主观性识别、倾向性判断、评测方法和语料建设等环节，不同方法在算法复杂度、效率和适用范围上存在差异[3]。因此，在本文研究中，应优先选择可实现、可解释、可测试的方法，避免单纯追求复杂模型而削弱系统整体链路的完整性。

从平台技术看，Spark 是面向大规模数据处理的计算框架，适合完成批量数据的统计分析与迭代计算。早期 MapReduce 模型为大规模数据处理提供了简洁的编程抽象，但其基于非循环数据流的计算模型对需要复用中间数据的迭代任务并不完全适合[4]。Spark 在此基础上提出了面向工作集复用的计算框架，可通过 RDD 等抽象支持迭代计算和交互式查询[5]。将 Spark 引入新闻热度监测场景，有助于在单机实验条件下验证批量文本处理链路，并为后续扩展到更大规模新闻数据处理奠定基础。

## 1.2 研究目的与意义

本课题的研究目的在于设计并实现一个面向新闻文本数据处理场景的热度监测系统，重点解决新闻来源异构、原始文本噪声较多、重复转载干扰统计结果以及分析结果缺少统一展示载体等问题。系统通过对新闻数据进行统一采集、标准化清洗、重复内容识别、热度指标计算和结果可视化展示，形成完整的分析闭环。系统实现时，将本地数据集导入、RSS 采集和网页正文解析作为数据入口，将清洗去重作为数据质量保障，将 Spark 批处理作为分析核心，将 Flask 接口和 Vue3 页面作为结果展示载体。

本课题的研究意义主要体现在三个方面。第一，从工程实践角度看，系统验证了 Spark 在新闻文本批处理场景中的可行性，并实现了从采集到展示的完整技术链路。第二，从方法应用角度看，系统将新闻采集、文本清洗、热度计算、关键词趋势分析、情感统计与事件聚类有机结合，提升了监测结果的可解释性。第三，从工程训练与系统研究角度看，该系统具备清晰的模块划分、稳定的运行流程和可量化的测试结果，可为新闻文本分析类系统的设计与实现提供参考案例。

与单纯的新闻查询工具相比，本系统更强调新闻热度形成过程的分析。系统不仅展示某条新闻本身，还关注某类关键词在不同时间段内的出现频率、不同来源对热点的覆盖程度、新闻情感倾向的统计分布以及相似新闻能否聚合为同一事件。网络舆情事件链相关研究指出，舆情事件通常会随时间发展衍生出多个关联事件，通过事件提取和演化分析可以帮助用户理解事件发展过程[6]。本课题虽然不深入构建复杂事件链模型，但通过事件聚类和趋势展示，为后续事件演化分析保留了扩展基础。

## 1.3 国内外研究现状

国内外关于新闻监测、舆情分析和热点发现的研究主要集中在数据采集、文本预处理、主题检测、情感分析和可视化呈现等方向。国外研究起步较早，较多工作关注新闻流数据处理、事件检测与主题演化建模，并在大规模文本分析框架和流式处理技术方面积累了较为成熟的成果。主题模型方面，Blei 等提出的 LDA 模型为离散文本集合的主题建模提供了经典概率模型基础，在新闻话题发现和文本聚类研究中被广泛使用[7]。大数据处理方面，MapReduce 和 Spark 等框架推动了批量文本数据的并行处理能力，为新闻数据的规模化分析提供了技术支撑[4，5]。

国内研究更强调中文语境下的新闻舆情监测、热点事件识别和行业应用。中文文本情感分析研究对情感识别、信息抽取、应用场景和研究难点进行了较多总结，认为情感分析是中文信息处理和网络文本挖掘中的重要方向[2，8]。近年的文本情感分析综述进一步梳理了深度学习、预训练模型和多模态分析等进展，同时也指出不同场景下仍需要关注数据质量、领域适配和可解释性问题[9]。在网络舆情分析方面，相关研究开始从单一话题热度统计转向事件链、演化路径和细粒度事件关系分析，以便更好地描述舆情随时间变化的过程[6]。

现有研究为本课题提供了重要参考，但在本科毕业设计场景下仍存在两个明显问题。其一，很多研究更关注模型精度或平台规模，而对单机环境下完整业务链路的构建讨论相对较少。其二，不少系统强调算法本身，却忽略了新闻数据质量控制，导致热度统计和聚类结果容易受到重复转载和脏数据影响。因此，本课题在参考现有研究成果的基础上，更突出“数据底座优先、结果可解释、系统链路完整”的实现思路。系统在算法选择上采用关键词统计、基础情感分析和轻量事件聚类等方法，以保证结果具有可运行性与可解释性；在工程实现上采用 Spark 批处理、Flask 接口和 Vue3 可视化页面，以保证系统具有较好的测试性与扩展性。

## 1.4 研究内容与论文结构

本课题的核心研究内容包括以下几个方面。第一，构建多来源新闻采集机制，支持本地数据集、RSS 源和网页正文解析等数据入口。第二，设计新闻数据清洗与去重流程，对原始新闻中的 HTML 噪声、异常时间格式、缺失字段和重复转载进行处理。第三，基于 Spark 完成热点统计、关键词趋势、基础情感分析和事件聚类等批处理任务。第四，设计后端 API 接口，实现新闻查询、热点列表、趋势数据、事件聚类和数据导出等服务能力。第五，使用 Vue3 和 ECharts 构建可视化页面，使系统分析结果能够以图表、列表和筛选条件形式呈现。ECharts 作为开源 Web 可视化框架，支持丰富图表类型、交互配置和高性能渲染，适合本系统的可视化展示需求[10]。

围绕上述内容，全文共分为六章。第一章介绍课题背景、研究意义、研究现状及论文结构；第二章分析系统相关技术与功能需求；第三章给出系统总体架构和各功能模块设计；第四章阐述系统实现过程；第五章对系统进行功能测试与结果分析；第六章总结全文并提出后续优化方向。

# 2 相关技术与需求分析

本章主要从系统实现所需的关键技术和业务需求两个方面展开论述。技术部分重点说明 Spark、Flask、Vue3、ECharts、新闻采集与文本处理相关工具在本系统中的作用；需求部分从功能、性能和可扩展性等角度明确系统建设目标，为后续总体设计与实现提供依据。本章不直接展开系统实现代码，而是说明技术选型依据以及各项技术在系统中的具体作用。

## 2.1 相关技术分析

本系统属于典型的数据驱动型应用，需要同时兼顾数据处理能力、服务封装能力和结果展示能力。为保证系统能够完成新闻文本分析任务并保持实现路径清晰，系统采用 Python 作为主要开发语言，并结合 Spark、Flask 和 Vue3 构建前后端一体化方案。系统技术选型遵循三个原则：一是工具成熟稳定，便于开发和部署；二是与新闻文本处理场景匹配，能够支持批量分析和可视化展示；三是结构清晰，便于测试验证与后续维护。

### 2.1.1 Spark 与 PySpark

Spark 是本系统的核心分析引擎，主要用于批量新闻数据的结构化处理、聚合统计和分析结果生成。Spark 相较于传统批处理框架的优势在于支持内存计算、工作集复用和较灵活的数据抽象，适合需要多次迭代或多维聚合的数据分析任务[5]。虽然本文的实验环境采用单机部署，但新闻热度监测任务本身具有明显的批量处理特征，例如对大量新闻标题和正文进行清洗、按时间窗口统计关键词出现次数、按来源统计新闻覆盖情况以及按相似度合并事件文本等。

PySpark 是 Spark 面向 Python 语言提供的接口，能够让开发者使用 Python 编写 Spark 分析任务。对于本系统而言，PySpark 的优势主要体现在两点。第一，Python 生态中有较多文本处理、数据清洗和机器学习工具，便于与 Spark 分析流程结合。第二，系统后端同样采用 Python 技术栈，使用 PySpark 能够降低跨语言集成成本。在本项目中，Spark 的使用重点不是搭建复杂分布式集群，而是在单机模式下验证面向批量新闻处理的技术路线，使系统具备后续扩展能力。

### 2.1.2 Flask 后端框架

Flask 负责承载系统的业务服务接口，向前端页面提供统计概览、热点列表、趋势查询、事件分析和数据导出等能力。新闻热度监测系统的后端服务不需要复杂的企业级权限体系和微服务拆分，更需要一个轻量、清晰、易维护的接口层。因此，采用 Flask 可以在较短开发周期内完成 API 封装，并与数据处理脚本、SQLite 数据库和导出功能形成较紧密的工程组合。

在本系统中，Flask 的主要作用不是完成新闻分析计算，而是作为分析结果和前端页面之间的桥梁。Spark 或 Pandas 任务负责生成热点、趋势、情感和事件聚类等结果；Flask 负责读取这些结果，并按接口规范返回给前端。这样的分工能够避免分析逻辑和页面逻辑相互耦合，也便于后续把离线分析脚本扩展为定时任务。

### 2.1.3 Vue3 与 ECharts 可视化技术

前端部分采用 Vue3 构建页面交互逻辑，采用 ECharts 实现图表展示。新闻热度监测系统的分析结果具有明显的数据可视化需求，例如热点榜单适合使用表格或排行展示，关键词趋势适合使用折线图或柱状图，情感分布适合使用饼图或环形图，事件聚类结果适合使用列表和标签形式呈现。ECharts 论文指出，其目标是提供开源、跨平台、支持交互式 Web 可视化的框架，并通过声明式配置方式降低图表构建复杂度[10]。因此，Vue3 与 ECharts 的组合能够较好满足本系统“数据驱动页面展示”的需求。

在系统实现中，Vue3 负责页面组织、组件状态和接口调用，ECharts 负责具体图表绘制。前端并不直接处理复杂分析算法，而是通过调用 Flask 接口获取后端处理结果，再把结果渲染为图表和列表。这种前后端分离方式有助于保持系统结构清晰，也有利于明确数据生产层与展示层之间的职责边界。

### 2.1.4 新闻采集与文本处理技术

在数据获取阶段，系统综合使用 RSS 解析、网页正文抽取和本地数据集导入等方式扩大数据来源范围。本地数据集用于保证基础样本量，RSS 源用于获取相对标准化的动态新闻，网页正文解析用于补充非标准新闻页面内容。三类来源共同构成混合采集方案，有助于降低单一数据源失效对数据连续性的影响。

在文本处理阶段，系统使用 HTML 清洗、时间标准化、关键词提取和文本相似度计算等方法提升数据质量。新闻网页中往往包含导航栏、广告、脚本、样式和推荐内容，如果直接进入分析流程，会干扰关键词统计和事件聚类结果。新闻转载现象也会使同一事件被重复统计，因此系统需要对 URL、标题和正文内容进行去重判断。文本预处理质量直接影响后续热度计算和聚类效果，是本系统实现中最基础也最关键的环节。

### 2.1.5 文本分析与聚类相关方法

为实现热点识别、趋势分析和事件聚类，系统采用关键词统计、时间窗口聚合、基础情感分析和 TF-IDF 向量化等方法构建分析流程。关键词统计能够反映某一时期内新闻文本中高频主题的变化；时间窗口聚合能够观察热点随时间的增强或减弱；情感分析能够从积极、中性和消极等维度辅助理解新闻文本倾向；事件聚类则用于将相似标题和摘要归并为同一事件。上述方法在实现复杂度与结果可解释性之间保持了较好的平衡，符合本文的研究范围与实现条件。

从研究基础看，LDA 等主题模型能够从文本集合中挖掘潜在主题，被广泛用于文本建模、话题发现和新闻主题分析[7]。但考虑到本系统需要优先保证功能闭环和运行稳定性，事件聚类部分暂不采用复杂主题模型作为核心算法，而采用 TF-IDF 表征和相似度聚类实现基础事件归并。该策略便于解释聚类依据，也便于通过标题和关键词人工核验聚类结果。

## 2.2 需求分析

新闻热度监测系统的核心目标不是单纯展示图表，而是建立“采集、分析、展示、导出”一体化业务链路。因此，本系统的需求分析应同时覆盖数据层需求和功能层需求，确保系统既有数据来源支撑，又能输出可观察、可查询、可验证的分析结果。结合本课题任务和实际实现条件，系统需求可以分为功能需求、性能需求、可用性需求和扩展性需求四类。

### 2.2.1 功能需求

系统应具备多来源新闻采集、原始数据存档、文本清洗与去重、热点榜单生成、关键词趋势统计、情感概览分析、事件聚类分析、新闻查询筛选和数据导出等基础功能。其中，采集与清洗模块负责提供可信数据底座，分析模块负责形成结构化结果，展示模块负责将结果转化为用户可读信息。

具体来说，新闻采集功能需要支持从本地文件、RSS 源和网页正文解析模块获取数据，并将不同来源的数据映射为统一字段。数据清洗功能需要去除 HTML 标签、标准化发布时间、处理空字段和异常字符。去重功能需要识别 URL、标题或正文相似的重复新闻，降低重复转载对热度统计的影响。热点分析功能需要按照报道量、时间因素和来源覆盖情况生成热点列表。趋势分析功能需要展示关键词随时间变化情况。情感分析功能需要给出新闻文本的基础情绪分布。事件聚类功能需要归并相似新闻，辅助用户理解同一事件下的多篇报道。查询与导出功能需要支持用户按关键词、来源和时间条件检索新闻，并导出筛选结果。

### 2.2.2 性能需求

系统需要在单机环境下保证基本的处理效率和响应能力。对于新闻批处理任务，系统应在可接受时间内完成数据清洗、统计与结果生成；对于查询与展示任务，系统应在常规交互场景下保持较稳定的接口响应和图表加载速度。考虑到本文实验环境以个人计算机为主，性能目标不应简单对标生产级舆情平台，而应重点验证系统在既定数据规模下的稳定性与结果生成能力。

本系统性能需求主要体现在三个方面。第一，批处理任务应能够处理一定规模的新闻样本，并输出稳定的分析结果。第二，后端接口在读取已生成结果时应保持较快响应，避免前端页面长时间等待。第三，前端图表在展示热点榜单、关键词趋势和情感分布时应保持清晰、稳定，不出现图表加载失败、数据错位或页面卡顿等问题。

### 2.2.3 可用性与可扩展性需求

为满足后续迭代与扩展需求，系统不仅要具备较好的可用性，还应保留一定扩展空间。可用性方面，系统页面应尽量保持结构清晰，用户能够通过导航进入可视化大屏、新闻查询、事件分析和服务中心等功能区域。查询条件和导出入口应便于理解，图表标题和数据指标应与后端返回字段保持一致。

可扩展性方面，系统应尽量通过配置文件管理新闻源，避免每增加一个来源都修改核心代码。分析模块应保持结果文件结构稳定，使后续替换更高质量的中文情感模型或事件聚类算法时，不影响前端页面调用。数据存储层应支持从 SQLite 扩展到 MySQL，分析层应支持从单机 Spark 逐步扩展到更大规模的数据处理环境。

## 2.3 可行性分析

从技术角度看，Python、Spark、Flask 和 Vue3 等技术组合成熟稳定，能够支撑本系统主要功能实现。Spark 用于批量分析，Flask 用于接口封装，Vue3 与 ECharts 用于前端展示，各技术之间职责划分清晰。从资源角度看，本项目采用单机部署方式，默认使用 SQLite 存储，开发和运行成本较低；当 Spark 环境不稳定时，系统可以使用 Pandas 兼容流程保持核心分析链路可用。从实现周期角度看，系统按“数据底座优先、分析能力逐步补充、展示页面同步联调”的策略推进，具有较强的可实施性。

从经济可行性看，本系统使用的主要技术和工具均为开源方案，不需要额外购买商业软件或专用硬件。系统运行所需数据可以来自公开新闻数据集、RSS 源和本地样例数据，能够支撑系统开发、实验验证和论文分析。从操作可行性看，系统前端提供可视化页面和查询导出入口，用户无需直接操作数据库或运行分析脚本即可查看热点结果。因此，本课题在既定研究周期内完成系统开发与论文撰写具有可行性。

# 3 系统设计

本章主要给出系统的总体设计思路、总体架构、功能模块划分和数据组织方式。其中，系统总体架构部分构成全文设计逻辑的主线，后续实现与测试均围绕该架构展开。

## 3.1 设计目标与原则

系统设计遵循“结构清晰、链路闭环、结果可解释、实现可落地”的原则。具体而言，一方面要保证系统覆盖新闻采集、清洗、分析、展示和导出的完整流程；另一方面要避免过度追求复杂模型和超大规模平台能力，从而保持系统结构稳定、逻辑清晰并便于验证分析结果。

## 3.2 系统总体架构

基于业务流程和数据流向分析，本文将新闻热度监测系统划分为数据源接入层、原始数据层、清洗去重层、Spark 批处理分析层、后端服务层和前端展示层六个部分。该架构既体现了新闻监测系统“先采集、再治理、后分析、再展示”的典型处理链路，也能够突出 Spark 在批量新闻文本分析中的核心作用。

从整体流程看，系统首先从本地数据集、RSS 源和网页正文解析模块获取原始新闻数据，并将采集结果按照统一字段结构保存到原始数据层。随后，系统对原始新闻进行 HTML 清洗、时间格式归一化、文本标准化以及强弱去重处理，形成适用于批处理分析的结构化新闻数据集。Spark 分析层在清洗结果基础上计算热点指数、关键词趋势、情感概览和事件聚类等结果，并将分析产物写入结构化结果文件或数据库。后端服务层进一步将分析结果封装为 REST 接口，供前端页面调用；前端展示层最终以可视化大屏、查询页面和事件分析页面形式完成用户交互。

这一六层架构的优点主要体现在三个方面。第一，数据采集、数据治理和结果分析职责分离，便于后续模块独立维护和扩展。第二，Spark 计算层位于清洗去重层之后，有助于提升统计结果的可信度，避免重复转载新闻对热度值造成放大。第三，前后端分离结构便于后续增加更多业务页面和分析能力，并有利于保持接口层与展示层的独立演进。

![图3.1 系统总体架构图](output/doc/figures/system_architecture.png)

### 3.2.1 数据源接入层

数据源接入层负责从不同来源收集新闻文本，是系统的数据入口。考虑到实验条件下不同数据来源可能存在稳定性差异，系统采用“本地数据集导入 + RSS 采集 + 网页正文解析”的混合策略。该设计既能保证样本规模，又能兼顾新闻数据的真实性与时效性。

### 3.2.2 原始数据层

原始数据层主要负责保存未经深入处理的新闻原始记录，包括标题、正文、摘要、来源、发布时间、抓取时间、链接地址及原始 HTML 等信息。保留原始数据的目的在于便于问题追踪、清洗规则校验和后续数据回溯，为系统调试和论文测试提供依据。

### 3.2.3 清洗去重层

清洗去重层是本系统保证数据质量的关键部分，主要完成 HTML 标签剥离、空白字符规范化、时间字段标准化、缺失值处理、强去重和弱去重等任务。其中，强去重主要依赖 URL 与标题哈希识别完全重复内容，弱去重则通过文本相似度或 SimHash 等方式识别转载和轻度改写新闻。该层的存在能够显著降低重复报道对热点统计的干扰。

### 3.2.4 Spark 批处理分析层

Spark 批处理分析层在清洗后的结构化新闻数据基础上完成核心分析任务，包括热点榜单生成、关键词趋势统计、情感结果汇总、事件聚类和预警规则输出等。热度指标设计采用可解释的综合加权思想，使报道量、来源多样性、时效性等因素能够共同参与排序，从而提升结果的业务解释性。

### 3.2.5 后端服务层

后端服务层负责整合数据处理结果并对外提供统一接口，主要功能包括概览统计查询、热点列表查询、关键词趋势查询、情感分析结果查询、事件聚类查询以及新闻数据导出等。该层使 Spark 产生的离线分析结果能够以 API 形式稳定输出，是连接分析层和展示层的重要桥梁。

### 3.2.6 前端展示层

前端展示层是系统与用户直接交互的部分，主要包含可视化大屏、新闻检索页面、事件分析页面和服务中心页面等。前端通过图表、列表和筛选组件展示系统分析结果，使抽象的数据统计转化为可观察、可操作的界面信息，从而提高系统的实用性与结果呈现能力。

## 3.3 功能模块设计

在总体架构基础上，系统进一步划分为新闻采集模块、数据清洗与去重模块、热点分析模块、趋势与情感分析模块、事件聚类模块、查询与导出模块六类核心功能模块。各模块之间围绕统一新闻数据结构进行协作，既相对独立，又能通过数据流形成完整闭环。

![图3.2 系统功能模块图](output/doc/figures/function_modules.png)

### 3.3.1 新闻采集模块

新闻采集模块负责从多种来源获取原始新闻数据，并对采集结果进行统一字段映射。该模块的设计重点在于来源配置可维护、采集流程可复用以及新增新闻源时代码改动最小化。

### 3.3.2 数据清洗与去重模块

该模块负责消除原始新闻中的噪声信息和重复记录，为后续统计分析提供可信数据基础。由于新闻转载和同源改写在实际场景中非常常见，因此该模块在系统设计中占据核心地位。

### 3.3.3 热点分析模块

热点分析模块通过对新闻报道量、来源覆盖度、时间分布等指标进行综合计算，形成热点榜单和概览统计结果。该模块是系统面向新闻热度监测任务的核心分析模块，也是前端大屏展示的主要数据来源。

### 3.3.4 趋势、情感与事件分析模块

该模块包括关键词趋势分析、情感统计和事件聚类三部分内容，主要用于从更细粒度角度呈现热点新闻的时间变化特征、情绪倾向特征和主题聚合特征。通过这部分设计，系统不再停留于简单的热点排序，而是具备更丰富的分析维度。

### 3.3.5 查询与导出模块

查询与导出模块面向新闻检索和结果复用需求，支持按关键词、来源、时间区间等条件对新闻数据进行筛选，并以 CSV 或 Excel 形式导出结果。该模块增强了系统的实用性，也便于后续开展人工核验与测试分析。

## 3.4 数据库与数据组织设计

为了兼顾开发效率与运行稳定性，系统默认采用 SQLite 作为数据存储方案，并保留切换到 MySQL 的兼容能力。数据库主要存储新闻基础信息、热点主题、关键词趋势、事件聚类和预警记录；中间处理结果则同时以 Parquet 和 JSON 文件形式保存，便于 Spark 任务读取、接口调试和测试结果复核。SQLite 适合单机运行和轻量部署，Parquet 文件适合保存结构化分析结果，JSON 文件则便于人工核验与接口联调。

系统核心数据表包括 `news_articles`、`hot_topics`、`keyword_trends`、`event_clusters` 和 `alert_records`。其中，`news_articles` 是系统最重要的基础表，保存清洗后的新闻明细及其分析字段；其他表主要保存批处理任务生成的聚合结果。该设计避免前端页面每次请求时重新执行复杂统计，能够提高接口响应效率，也使系统测试时可以直接核验各类结果表是否生成完整。

## 3.5 热度指标与分析流程设计

新闻热度监测的核心在于构建可解释的热度指标。如果仅按照新闻数量排序，重复转载多的新闻容易获得过高排名；如果只按照时间排序，则无法反映来源覆盖和情感强度。因此，本系统采用综合加权方法计算新闻热度，重点考虑报道量、情感强度、来源权威度、互动程度和时效性五类因素。该指标设计不追求复杂模型训练，而强调计算过程清晰、结果便于解释，符合毕业设计系统的实现目标。

在系统实现中，通用场景下热度得分由报道量得分、情感强度得分、来源权威度得分、互动得分和时效性得分加权得到。其中，报道量得分反映同一来源或同类主题中的新闻覆盖规模；情感强度得分反映文本正负向程度的绝对强度；来源权威度得分根据新闻来源预设权重计算；互动得分综合点赞、评论、转发和阅读等字段，在缺少真实互动数据时以关键词丰富度作为补充；时效性得分随新闻发布时间距离当前时间的增加而衰减。系统还保留危机场景和政策场景的权重方案，使不同业务场景可以强调不同指标。

分析流程方面，系统先对清洗后的新闻数据进行时间字段转换和情感计算，再执行热度得分计算。随后，系统按照关键词展开新闻记录，过滤低价值词，按日期和关键词进行分组聚合，生成热点主题和关键词趋势。事件聚类部分首先拼接标题和摘要形成事件文本，再使用 TF-IDF 将文本转换为向量，最后通过凝聚层次聚类生成事件簇。预警分析部分按小时窗口统计新闻数量、负面情感比例和平均热度，当新闻量明显高于平均水平或负面比例超过阈值时生成预警记录。

从相关研究脉络来看，中文文本分析在文本分类、观点挖掘、短文本聚合、关键词抽取和术语抽取等方向已经积累了较为系统的方法体系，这些研究为本系统的关键词统计、文本表示、相似度计算和事件归并提供了直接参考[11，12，13，14，15]。同时，并行计算相关综述研究也指出，大数据分析任务与并行化框架的结合能够有效提升数据处理效率和算法扩展能力[16]。因此，本文在设计新闻热度监测系统时，不再把各类分析任务割裂实现，而是将文本预处理、特征抽取、热点统计和聚类分析统一纳入批处理链路中。

## 4.1 开发环境与项目结构实现

本节说明系统采用的开发语言、主要依赖库、前后端目录结构和运行方式，为后续功能实现描述提供环境基础。

系统后端采用 Python 进行开发，主要目录包括 `backend/app`、`backend/scripts` 和 `backend/config`。其中，`backend/app` 负责 Flask 应用初始化、接口注册和数据库连接；`backend/scripts` 负责数据采集、初始化数据库、批处理分析和批量导入等离线任务；`backend/config/source_catalog.json` 用于维护新闻源配置，降低新增数据源时对业务代码的影响。前端部分采用 Vue3 和 Vite 构建，页面文件集中在 `frontend/src/pages` 目录下，公共 API 调用封装在 `frontend/src/services/api.js` 中。

从工程组织角度看，系统目录结构与第三章设计的分层结构基本一致。数据采集和分析脚本承担数据源接入层、清洗去重层和 Spark 分析层职责；Flask 应用承担后端服务层职责；Vue 页面承担前端展示层职责。这样的组织方式使论文中的架构描述与实际工程文件形成较为清晰的对应关系。

系统运行由后端服务和前端服务两部分组成。后端通过 Flask 应用加载配置、初始化数据库连接并注册接口蓝图；前端通过 Vite 启动开发服务或构建静态文件。为保证系统能够在无外部数据库服务的环境下运行，系统默认将 SQLite 数据库文件保存在 `thesis/data/news_monitor.db` 中，并将处理结果保存到 `thesis/data/processed` 目录。该组织方式将主要运行依赖集中在项目目录内，降低了环境部署复杂度。

## 4.2 新闻采集与数据治理实现

本节重点说明本地数据导入、RSS 采集、网页正文抽取、字段标准化、清洗规则和去重策略的实际实现方式，突出新闻数据底座建设在本系统中的核心地位。

系统采集模块首先读取新闻源配置或本地导入文件，将不同来源的新闻记录转换为统一结构。统一字段包括新闻编号、标题、正文、摘要、链接、来源、来源类型、发布时间、抓取时间、分类、地区、关键词和原始 HTML 等内容。统一字段结构的作用是屏蔽不同数据源之间的格式差异，使后续清洗、去重和分析任务可以按照同一数据模型执行。

在清洗阶段，系统主要对新闻正文中的 HTML 标签、脚本样式、异常空白符和特殊字符进行处理，并对发布时间字段进行标准化转换。由于新闻数据中经常存在标题相同、链接重复或正文高度相似的转载内容，系统在去重阶段采用强去重和弱去重结合的方式。强去重用于识别 URL 或标题哈希完全一致的记录，弱去重用于识别标题相似或正文相似的近重复新闻。经过该流程处理后，系统能够输出较为规整的新闻数据，为 Spark 批处理分析提供可靠输入。

![图4.1 新闻数据处理流程图](output/doc/figures/data_flow.png)

清洗实现中，系统首先使用 HTML 解析工具移除 `script`、`style`、导航栏、页脚等与新闻正文无关的标签，再将剩余文本转换为规整字符串。对于标题和摘要中常见的编辑来源、责任编辑、HTML 转义字符等噪声，系统通过正则规则进行剥离。时间字段使用统一的时间解析函数进行处理，遇到带时区的时间值时转换为无时区时间戳，避免 Pandas 或数据库写入时出现格式不一致问题。

去重实现分为两个阶段。第一阶段为强去重，主要依据 URL 或标题哈希判断完全重复内容。如果两条新闻链接相同，或者在缺少链接时标题哈希相同，则保留其中一条记录。第二阶段为弱去重，系统按照发布时间日期和标题前缀构造候选桶，只在同一候选桶内比较文本相似度，避免全量两两比较造成不必要的性能开销。弱去重综合使用 SimHash 距离和标题相似度，当两条新闻的 SimHash 距离较小或标题相似度较高时，将其视为近重复新闻。这样的实现能够兼顾处理效率和去重准确性。

## 4.3 Spark 批处理分析实现

本节将围绕热点统计、关键词趋势、情感概览、事件聚类和结果输出等功能，说明 Spark 任务在系统中的实现过程，并解释相关参数和指标设计思路。

Spark 批处理分析模块以清洗后的新闻数据为输入，按照不同分析目标生成多个结果文件或结果表。热点分析主要统计新闻在一定时间范围内的报道数量、来源覆盖情况和时间分布情况，并结合系统设定的热度指标进行排序；关键词趋势分析则按照关键词和时间窗口聚合新闻数量，反映不同关键词在时间维度上的变化趋势；情感分析部分对新闻文本进行基础情绪倾向统计，用于辅助判断热点事件的情绪变化；事件聚类部分通过标题、摘要和关键词信息识别相似新闻，将同一事件下的多条新闻记录归并为事件簇。

在本文实现中，Spark 的作用主要体现在批处理链路验证和可扩展能力说明上。系统当前以本地 PySpark 作为优先方案，当运行环境不满足 Spark 条件时，可退化为 Pandas 分析流程，以保持分析结果输出结构的一致性。该设计既体现了 Spark 在大数据场景下的适用性，也兼顾了实验环境的稳定性。

在具体实现上，系统首先将清洗后的记录转换为 DataFrame，并补充发布时间日粒度字段、小时粒度字段、互动总量字段和情感分析字段。情感分析部分采用词典匹配和规则修正方法，将新闻划分为正面、中性和负面三类，并给出情感强度值。虽然该方法不如深度学习模型复杂，但具有运行速度快、结果可解释、便于论文说明和测试复现等优点，适合作为本系统的基础情感分析方案。

热点主题计算时，系统将每条新闻的关键词字段展开为多行记录，过滤无意义词后按日期和关键词分组统计新闻数量、来源数量、平均热度和情感强度。系统要求热点关键词至少出现两次，避免偶发词进入热点榜单。关键词趋势计算则在最近时间窗口内筛选高频关键词，并按日期生成趋势数据，前端可根据该结果绘制折线图或柱状图。事件聚类计算时，系统将标题和摘要拼接为事件文本，使用 TF-IDF 向量化方法提取文本特征，并采用凝聚层次聚类算法进行归并。聚类结果会保存代表标题、关键词、新闻数量和聚类得分，供事件分析页面展示。自动文本分类、词项加权、近重复文本相似度估计以及经典信息检索研究，为本文选择 TF-IDF 表示、关键词排序和近重复新闻识别策略提供了理论支撑[17，18，19，20]。

预警分析实现基于小时窗口统计。系统按小时聚合新闻数量、负面情感比例、平均热度和来源数量，当某一小时新闻数量显著高于平均水平时生成热点突增预警；当负面情感比例超过设定阈值时生成负面倾向预警。预警等级划分为关注、预警、严重和紧急四类，便于页面和服务简报按照不同风险程度进行展示。

## 4.4 后端接口实现

本节主要说明 Flask 接口路由设计、核心接口返回结构、查询筛选逻辑以及数据导出实现方式，体现系统服务层的工程实现过程。

后端服务层围绕前端展示需求提供接口，主要包括概览统计、新闻检索、热点列表、关键词趋势、情感统计、事件聚类和新闻导出等接口。接口返回结果统一采用 JSON 格式，便于前端使用 Axios 进行请求并交由 ECharts 或列表组件进行展示。对于新闻检索接口，系统支持按关键词、来源和时间条件进行筛选；对于导出接口，系统可将筛选后的新闻结果导出为 CSV 或 Excel 文件，方便后续分析与论文测试记录整理。

后端接口实现的关键在于保持接口职责清晰。分析任务不直接耦合到页面逻辑中，而是先由采集与分析脚本生成结果，再由 Flask 读取结果并提供查询服务。这样可以降低前端页面与数据处理脚本之间的耦合度，也便于后续将离线批处理任务替换为定时任务或流式处理任务。

表4.1 后端主要接口设计

| 接口地址 | 请求方式 | 功能说明 | 返回内容 |
| --- | --- | --- | --- |

测试过程中还观察到，Spark 在单机环境下存在较为明显的初始化开销，尤其在千级新闻样本条件下，启动成本在总耗时中占比较高。为降低运行环境差异对结果生成的影响，系统保留了 Pandas 兼容分析流程；当 Spark 运行条件不足或启动异常时，系统可切换至兼容流程并保持结果输出结构一致。该设计体现了系统在分析框架选择上的兼容性，也提高了不同运行环境下结果生成的稳定性。

## 5.4 测试结果分析

综合功能测试和性能测试结果，系统完成了新闻采集、数据清洗、批处理分析、接口服务、前端展示和数据导出的完整流程。从功能完整性看，系统已经覆盖新闻热度监测的主要业务环节，并可通过热点榜单、趋势图表、情感分布和事件聚类等形式展示分析结果。从数据结果看，系统已生成多类结构化分析产物，并通过数据库与文件两种方式进行保存，便于后续复核与扩展。从展示效果看，可视化大屏、新闻查询页面和事件分析页面能够较完整地呈现分析结果及其来源。

系统测试也揭示出若干仍需改进的问题。第一，新闻源虽然覆盖本地数据、RSS 和开放接口，但不同来源的数据质量差异较大，部分来源存在摘要较短、正文缺失或时间格式不统一的问题。第二，情感分析主要采用词典和规则方法，虽可完成基础统计任务，但对隐含情绪、反讽表达和领域词汇的识别能力仍然有限。第三，事件聚类使用标题和摘要进行相似度计算，能够归并部分相似新闻，但对跨时间长周期事件演化的刻画仍不充分。第四，系统目前以批处理为主，预警功能仍基于统计窗口生成，尚未形成实时流处理机制。

总体来看，测试结果表明系统已经实现预定的研究与实现目标，既能够稳定运行并输出完整分析结果，也为后续扩展新闻源、优化模型和增强实时处理能力保留了进一步完善空间。后续结论部分将在此基础上对本文成果与不足作进一步总结。

# 6 结论与展望

本章对全文工作进行总结，并结合系统测试结果分析当前系统存在的不足和后续优化方向。通过前文论述可以看出，本课题围绕新闻热度监测任务，完成了从需求分析、系统设计、系统实现到测试验证的完整过程。系统以新闻数据为基础，以清洗去重为质量保障，以 Spark 批处理分析为核心，以 Flask 接口和 Vue3 可视化页面为展示载体，实现了具有完整闭环的新闻热度监测系统。

## 6.1 研究成果总结

本文完成的主要工作可以概括为以下几个方面。第一，完成了新闻热度监测系统的需求分析和总体架构设计，明确了数据源接入层、原始数据层、清洗去重层、Spark 分析层、后端服务层和前端展示层六层结构。第二，设计并实现了多来源新闻采集和数据治理流程，能够对本地数据、RSS 数据和开放 API 数据进行统一字段映射，并通过 HTML 清洗、时间标准化、强去重和弱去重提升数据质量。第三，基于批处理思想实现了热点主题、关键词趋势、情感统计、事件聚类和预警记录等分析功能，使系统不仅能够展示新闻列表，还能够从多个维度解释新闻热度变化。第四，采用 Flask 和 Vue3 实现了前后端分离展示，完成可视化大屏、新闻查询和事件分析等页面，使分析结果能够以图表和列表形式直观呈现。第五，通过功能测试和结果规模统计验证了系统功能完整性和运行稳定性。

从工程价值看，本系统将大数据处理思想应用于新闻文本分析场景，形成了采集、处理、分析、展示和导出的完整链路。从研究写作价值看，系统结构清晰、模块边界明确、测试结果具有可复核性，能够支撑需求分析、概要设计、详细设计、编码实现和系统测试等内容的系统展开。从应用价值看，系统可为新闻热点观察、舆情初步研判和事件聚合分析提供基础工具，也为后续引入更复杂的情感模型和事件演化模型提供工程基础。

## 6.2 存在的不足

虽然系统已经实现了新闻热度监测的主要功能，但仍存在一些不足。首先，新闻源覆盖范围仍有提升空间。当前系统虽然支持多来源采集，但部分数据源受网络访问、接口稳定性和页面结构变化影响，采集结果可能存在波动。其次，数据质量治理仍然依赖规则和相似度方法，对复杂转载、改写新闻和跨平台重复事件的识别能力有限。再次，情感分析方法较为基础，主要依赖词典和规则判断，面对复杂语义、行业专业表达和上下文情绪变化时准确率仍有不足。最后，事件聚类和预警功能仍属于基础实现，能够完成相似新闻聚合和简单阈值预警，但对事件生命周期、传播路径演化和实时响应机制的支持还不够深入。

## 6.3 未来展望

针对上述不足，后续可以从以下几个方向继续完善。第一，扩展新闻源配置和采集适配器，增加更多权威媒体、行业媒体和社交平台来源，并为不同来源设置更细粒度的数据质量评分。第二，引入更高质量的中文文本分析模型，例如基于预训练语言模型的情感分类、命名实体识别和主题识别方法，以提升复杂文本场景下的分析效果。第三，加强事件演化分析，在现有事件聚类基础上进一步构建事件时间线、传播路径和阶段识别模型，使系统能够从“发现热点”扩展到“解释热点如何发展”。第四，增强实时处理能力，后续可将 Spark Structured Streaming 或消息队列引入系统，使新闻采集、分析和预警更加接近实时。第五，扩展多角色服务场景，根据媒体编辑、企业公关、政府舆情管理和研究人员等不同用户需求提供差异化分析视图。

总体而言，基于 Spark 的新闻热度监测系统已经完成本文设定的核心研究目标。后续若继续优化数据源、算法模型和实时处理能力，系统有望进一步发展为面向实际业务场景的新闻热点分析平台。
