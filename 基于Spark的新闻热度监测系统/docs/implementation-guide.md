# 实施方案与技术操作

## 1. 我会怎么做这个项目

整体实施思路是“先搭底座，再出结果，最后做展示”，而不是一开始就陷入模型调优和复杂页面开发。

具体顺序如下：

1. 定义统一新闻数据结构
2. 建立采集模块，保证能收集真实新闻数据
3. 建立清洗去重模块，解决乱码、HTML 噪声、重复转载等问题
4. 将清洗结果保存为 Parquet，交给 Spark 做批处理
5. 输出热度、关键词趋势、情感、聚类结果
6. 通过 Flask 将结果封装为查询接口
7. 用 Vue + ECharts 搭建可视化大屏和查询导出界面

## 2. 关键技术操作

### 2.1 新闻采集

新闻采集采用三类策略：

- 本地数据集导入：用于保证基础样本量
- RSS 采集：适合快速接入标准新闻源
- 网页正文解析：用于补充非标准网站

当前工程已补充两种落地能力：

1. `backend/config/source_catalog.json` 统一维护真实新闻源配置
2. `data/raw/imports/` 支持批量投放本地新闻数据集文件，由脚本自动扫描导入

统一输出字段：

`news_id、title、content、summary、url、source、source_type、publish_time、crawl_time、author、category、region、keywords、raw_html、lang`

### 2.1.1 真实新闻源配置方式

RSS 和网页源都采用 JSON 配置，字段建议包含：

- `name`
- `url`
- `category`
- `region`
- `lang`
- `enabled`

这样做的好处是后面新增来源时不需要改代码，只需要补配置。

### 2.1.2 批量导入方式

本地批量导入支持三种格式：

- `csv`
- `json`
- `jsonl`

默认脚本会递归扫描 `data/raw/imports/`，把所有可识别数据文件统一导入，再进入清洗、去重和分析链路。

### 2.2 数据清洗

清洗流程包含：

1. 去除 HTML 标签与脚本样式
2. 标准化空白符和特殊字符
3. 统一时间格式
4. 处理缺失字段
5. 生成标题哈希和内容哈希
6. 提取关键词

### 2.3 去重策略

去重分为两层：

- 强去重：按 URL 和标题哈希直接删除完全重复新闻
- 弱去重：通过 SimHash 和标题相似度识别转载、改写和近重复新闻

设计这个模块的原因是新闻转载很多，如果不去重，热点指数会被虚高。

### 2.4 Spark 批处理

Spark 的输入是清洗后的 Parquet 文件，输出主要有：

- `news_cleaned`
- `hot_topics`
- `keyword_trends`
- `sentiment_results`
- `event_clusters`
- `alerts`

Spark 主要处理：

1. 新闻总量统计
2. 关键词分布统计
3. 不同来源和时间窗的热点聚合
4. 热度指数计算

当前工程默认先以 Pandas 跑通开发链路，避免 Windows 本地 Spark 环境不稳定影响联调。待本机 Spark 修复后，只需开启 `ENABLE_SPARK=1` 即可切换为本地 PySpark 模式。

### 2.5 热度指数设计

热度指数采用可解释的加权公式：

`热度 = 报道量权重 + 来源多样性权重 + 媒体权威度权重 + 情感强度权重 + 时效衰减权重`

这么设计的好处是：

- 容易解释
- 容易调参
- 适合毕业设计答辩
- 不依赖难采集的互动数据

### 2.6 情感分析

第一版优先使用基础情感分析能力：

- 当前工程内置词典规则法，确保系统可以跑通
- 后续可以接入公开中文情感分类模型
- 再后续可以用采集新闻样本做小规模补标和微调

这样做的好处是先保证系统可运行，再逐步提升精度。

### 2.7 事件聚类

事件聚类采用轻量实现：

1. 将标题和摘要拼接成事件文本
2. 进行 TF-IDF 向量化
3. 使用聚类算法将相似新闻归入同一事件
4. 用关键词提取结果给聚类命名

答辩时可以说明：第一版先验证事件聚合思路，后续可替换为中文句向量+BERT 聚类方案。

### 2.8 前后端联动

后端提供以下接口：

- `/api/dashboard/overview`
- `/api/news/search`
- `/api/hot/list`
- `/api/trends/keywords`
- `/api/sentiment/summary`
- `/api/events/clusters`
- `/api/export/news`

前端页面分为：

1. 可视化大屏
2. 新闻查询页
3. 事件分析页

## 3. 你在复试/答辩时怎么讲

### 如果老师问：为什么把采集放在最前面？

可以回答：

“因为新闻热度监测的结果质量由数据底座决定。如果采集数据不真实、内容不完整、转载重复没有处理，那么后面的热度分析、情感分析和聚类分析都会失真。所以我优先搭建采集、清洗和去重模块，把这部分作为系统的核心。” 

### 如果老师问：为什么用了 Spark？

可以回答：

“这个项目面对的是新闻文本批量处理场景，需要做结构化清洗、关键词聚合和多维统计。Spark 在本地模式下就能完成批处理验证，也方便后续扩展到更大规模的数据环境，所以我采用它作为分析引擎。” 

### 如果老师问：为什么没有一开始做复杂模型？

可以回答：

“毕业设计第一阶段要优先保证功能闭环和结果可解释，所以我先采用轻量方案实现情感分析和事件聚类。在系统整体跑通后，再通过公开中文数据集和补充标注做模型优化，这样开发风险更可控。” 

### 如果老师问：你这个系统的创新点是什么？

可以从三个角度回答：

1. 以新闻数据质量为核心，把采集、清洗、去重放在首位
2. 用可解释的热度指数把多维因素合并，便于系统展示和答辩说明
3. 在单机条件下完成“采集-分析-展示-导出”闭环，具备后续扩展到实时监测的基础

## 4. 实际开发建议

1. 先用样例数据把前后端和 API 全跑通
2. 再补 RSS 源和本地数据集，逐步累积真实数据
3. 每增加一个分析能力，就同步在前端加一个可视化结果
4. 每个阶段都保留截图、表格和测试记录，为论文和答辩做证据
